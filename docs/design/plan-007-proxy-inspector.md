## Overview

Build a standalone Electron desktop application (`proxy-inspector/`) that opens NDJSON log files generated by the NotebookLM Agent proxy, displays interactions grouped by `interactionId`, and live-tails the file as new entries are appended. The app uses electron-vite with React + TypeScript for the renderer, `fs.watch` with byte-offset tailing for file watching, and react-window v2 for virtualized scrolling.

## Phase 2: Main Process — File I/O, NDJSON Parsing, File Watching

**Goal**: Implement the complete main process: file dialog, NDJSON parsing, event grouping by `interactionId`, byte-offset file tailing with `fs.watch`, recent files list, and all IPC handlers.

### Files to Create

| File | Purpose |
|------|---------|
| `proxy-inspector/src/main/index.ts` | Extend: add IPC handlers, menu, window state persistence |
| `proxy-inspector/src/main/file-watcher.ts` | `fs.watch` + byte-offset incremental read, debounced at 500ms |
| `proxy-inspector/src/main/ndjson-tail-parser.ts` | Line splitter with remainder buffer for partial lines |
| `proxy-inspector/src/main/ndjson-grouper.ts` | Parse LogEntry[], group by interactionId, derive InteractionSummary |
| `proxy-inspector/src/main/recent-files.ts` | Persist last 10 opened files to `~/.proxy-inspector/recent.json` |
| `proxy-inspector/src/main/interaction-store.ts` | In-memory store: Map<interactionId, LogEntry[]> for on-demand payload fetch |

### Key Design Decisions

- **Event-type agnostic watcher**: On macOS, `fs.watch` always fires `'rename'`, not `'change'`. The handler ignores the event type and always calls `statSync` to check actual file size (per research doc).
- **Debounce at 500ms**: Matches the proxy's flush interval. Collapses burst events into one read.
- **False positive guard**: `if (stat.size <= bytesRead) return` — no unnecessary file reads.
- **Partial line handling**: `NdjsonTailParser` holds a remainder buffer. Lines split by `\n`; last element saved as remainder.
- **Malformed line tolerance**: `try/catch` around `JSON.parse`; skip with `console.warn`.
- **Summary vs full payload**: Initial `file-data` IPC sends only `InteractionSummary[]` (not full payloads). Full payloads fetched on-demand via `get-payload` IPC when user selects an interaction.

### IPC Handlers (registered in main/index.ts)

| Channel | Direction | Pattern | Purpose |
|---------|-----------|---------|---------|
| `open-file` | Renderer -> Main | invoke/handle | Show file dialog, parse file, start watcher, send summaries |
| `open-recent` | Renderer -> Main | invoke/handle | Open a recent file by path |
| `get-payload` | Renderer -> Main | invoke/handle | Return full LogEntry[] for a specific interactionId |
| `pause-watch` | Renderer -> Main | invoke/handle | Pause/resume file watcher |
| `get-recent` | Renderer -> Main | invoke/handle | Return recent files list |
| `file-data` | Main -> Renderer | send/on | Push initial InteractionSummary[] + file metadata |
| `new-events` | Main -> Renderer | send/on | Push incremental InteractionSummary[] updates |

### Acceptance Criteria

- Opening a file dialog and selecting the sample log file parses all 12 lines into 2 interactions.
- `InteractionSummary` for interaction #2 contains: userMessage "list my notebooks", roundTripCount 2, totalTokens 20773, durationMs 19068, toolCalls ["list_notebooks"], status "complete".
- `get-payload` for interaction #2 returns 8 LogEntry objects in chronological order.
- Appending a new NDJSON line to the file triggers `new-events` IPC within 2 seconds.
- Malformed lines are skipped with a warning (no crash).
- Recent files list persists across app restarts.
- Closing the watcher with `stop()` removes all timers and the `fs.watch` handle.

## Phase 3: Shared Types

**Goal**: Define the IPC contract, interaction summary types, and API interface shared between main, preload, and renderer.

### Type Definitions

```typescript
// types.ts
export type InteractionStatus = 'complete' | 'in-progress' | 'error';

export interface InteractionSummary {
  id: string;                    // interactionId
  index: number;                 // 1-based sequential number
  userMessage: string;           // from interaction_start payload (first 100 chars)
  timestamp: string;             // ISO-8601 from interaction_start
  status: InteractionStatus;
  durationMs: number | null;     // from interaction_end, null if in-progress
  roundTripCount: number;
  totalPromptTokens: number;
  totalCompletionTokens: number;
  totalTokens: number;
  toolCalls: string[];           // tool names from interaction_end
  hasErrors: boolean;            // contains llm_error or tool_error events
  eventCount: number;            // total events in this interaction
}

export interface FileMetadata {
  filePath: string;
  sessionId: string;             // extracted from filename
  createdAt: string;             // extracted from filename or file stat
  fileSize: number;
}

export interface ParsedFileData {
  metadata: FileMetadata;
  interactions: InteractionSummary[];
  aggregates: {
    totalInteractions: number;
    totalTokens: number;
    totalToolCalls: number;
    timeSpanMs: number;          // first to last event timestamp
  };
}

export interface IncrementalUpdate {
  interactions: InteractionSummary[];  // new or updated summaries
}
```

### IPC API Interface

```typescript
// ipc-types.ts
export interface ProxyInspectorAPI {
  openFile: () => Promise<ParsedFileData | undefined>;
  openRecent: (filePath: string) => Promise<ParsedFileData | undefined>;
  getPayload: (interactionId: string) => Promise<LogEntry[]>;
  pauseWatch: (paused: boolean) => Promise<void>;
  getRecentFiles: () => Promise<string[]>;
  onFileData: (cb: (data: ParsedFileData) => void) => () => void;
  onNewEvents: (cb: (update: IncrementalUpdate) => void) => () => void;
}
```

- `IPC` channel constants are importable from `@shared/ipc-channels`.
- `Window.api` is typed as `ProxyInspectorAPI` in the renderer (via env.d.ts).

### Payload Rendering Details

**LlmRequestPayload** (FR-5.2):
- Model name (prominent, non-collapsible)
- Contents count and conversation history (collapsible). Each content item shows role + text/functionCall/functionResponse. Mini chat bubble layout.
- System instruction text (collapsible, monospaced, potentially very long)
- Tool names as chip/tag list (non-collapsible)
- Tool declarations (collapsible, only on RT1, rendered as formatted JSON tree)

**LlmResponsePayload** (FR-5.3):
- Response text (handle both `{role, parts: [{text}]}` and `{role, parts: [{functionCall}]}` formats)
- Function calls: tool name + args as formatted JSON
- Token usage: prompt / completion / total with small badge
- Duration, streamed flag, chunk count, finish reason

**Lazy rendering** (NFR-1.3): Payload detail is only fetched and rendered when the user clicks an event in the timeline. Full LogEntry[] for the interaction is fetched via `window.api.getPayload(interactionId)` on first selection; cached in component state thereafter.

## Risk Mitigation

| Risk | Mitigation |
|------|-----------|
| electron-vite ESM/CJS interop issues | Fall back to manual esbuild scripts if electron-vite config proves intractable |
| react-window v2 API discrepancies from docs | Pin to v2.2.7; verify `onRowsRendered` and `scrollToRow` signatures empirically |
| Large payload rendering frame drops | Lazy render with collapsible sections; only fetch full payload on user click |
| fs.watch duplicate events on macOS | 500ms debounce; statSync guard for false positives |
| Memory with 100MB files | Send only summaries to renderer; full payloads on-demand; consider pagination for 10K+ interactions |
